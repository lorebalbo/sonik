# 1. PRD-0052: Mixer State Schema & ValueTree Integration

## 1.1. Problem

EPIC-0007 will introduce a full mixing console (per-channel gain, 3-band EQ with kills, filter, channel fader, A/B assign, crossfader, master gain, metering) that must be observed by the UI, driven by MIDI controllers, and consumed by the audio thread. Today there is no agreed data contract: the deck state tree (PRD-0001) only models per-deck transport / pitch / deck-gain fields, and the MIDI binding registry (PRD-0042) plus mapping storage (PRD-0043) reference mixer target identifiers (`mixer.channel.{A,B,C,D}.*`, `mixer.crossfader`, `mixer.master.gain`) for which no backing state exists. Without a shared schema, every sibling Mixer PRD (DSP, UI, MIDI feedback, persistence) would invent its own representation, the UI would be forced to poll the audio thread for parameter and meter values, and MIDI mapping would resolve to dangling identifiers. In addition, the registry already lists most channel-strip controls but is missing the per-channel `filter` target that this Epic introduces, so a community mapping referencing `mixer.channel.A.filter` would be rejected as `UnknownTarget`.

## 1.2. Objective

The system provides a single, authoritative mixer state schema, hosted as a `mixer` sub-tree of the central `juce::ValueTree` established by PRD-0001, such that:
- Every parameter listed in EPIC-0007 §1.3.2 exists as a typed, range-bounded, default-initialised `ValueTree` property at a stable, documented identifier path.
- UI components, the MIDI inbound router (PRD-0044), the MIDI feedback engine (PRD-0047), and future persistence layers all read and write mixer state through this single source of truth, without polling the audio thread.
- The audio thread reads continuous parameters via a parallel `std::atomic` snapshot block (one atomic per scalar parameter) updated by the UI / MIDI layer when the corresponding `ValueTree` property changes, and writes metering values into a separate lock-free atomic snapshot block that the UI reads at its own refresh rate without generating `ValueTree` listener traffic.
- PRD-0042 (Control Target Registry) and PRD-0043 (Mapping Storage) are amended to declare the new `mixer.channel.{A,B,C,D}.filter` target as Continuous with soft-takeover, eliminating the current `UnknownTarget` gap and giving every channel-strip control a registered binding target.
- The data contract is stable enough that every later Mixer PRD (PRD-0053…PRD-0061) and every later MIDI mapping file refers to these identifiers by name, with no schema migration required.

## 1.3. Developer / Integration Flow

This PRD is foundational and non-user-facing. The flow describes how downstream code consumes the schema.

1. At application startup, `Source/Features/Mixer/State/` creates the `mixer` sub-tree under the root `ValueTree`, populates `mixer.master`, `mixer.crossfader`, and four per-channel sub-trees `mixer.channel.A`, `mixer.channel.B`, `mixer.channel.C`, `mixer.channel.D` with the defaults listed in §1.4, and constructs the parallel `MixerAtomicSnapshot` block (one `std::atomic<float>` or `std::atomic<bool>` per scalar parameter, indexed by a stable enum mirroring the identifier path).
2. A `MixerStateBridge` registers as a `juce::ValueTree::Listener` on the `mixer` sub-tree. On every property change it writes the new value into the matching atomic slot in `MixerAtomicSnapshot`. This is the only writer to those atomics from the message thread.
3. UI components (introduced by PRD-0059 / PRD-0060) attach `juce::ValueTree::Listener`s scoped to the channel or master sub-tree they render, and update their visual state on `valueTreePropertyChanged`. UI never reads from `MixerAtomicSnapshot` and never polls the audio thread.
4. The audio engine (PRD-0053) reads parameter values exclusively via `MixerAtomicSnapshot::loadRelaxed(slotId)` from inside `processBlock`. It never touches the `ValueTree`.
5. The audio engine writes metering values (peak L/R, RMS L/R, clip-latched flag) for each channel and the master bus into a separate `MixerMeterSnapshot` block of atomics, also indexed by enum slot. Metering values do not live in the `ValueTree`. UI meter components poll `MixerMeterSnapshot` from a `juce::Timer` at the UI refresh rate (e.g. 30–60 Hz) defined by PRD-0058.
6. The MIDI inbound router (PRD-0044) resolves an incoming MIDI message to a `ControlTarget` identifier via PRD-0042's registry. For `mixer.*` targets it writes the resolved, normalised value back into the `mixer` `ValueTree` property at the matching path; the `MixerStateBridge` in step 2 then propagates the change into the atomic snapshot. The MIDI feedback engine (PRD-0047) reads from the same `ValueTree` paths to emit outbound LED state.
7. When a deck is added or removed at runtime (EPIC-0001 dynamic deck count, 1–4 decks), the deck's mixer channel sub-tree is kept in place and reset to defaults on removal (see §1.5.5), and re-used as-is on re-addition. The sub-tree is never destroyed during the application lifetime.
8. When the application shuts down, the `mixer` sub-tree is the canonical serialisation surface. Persistence is out of scope for this PRD but is unblocked by it.

## 1.4. Acceptance Criteria

- [ ] The central `juce::ValueTree` (PRD-0001) exposes a child sub-tree with identifier `mixer`, containing exactly the children documented in §1.3.2 of EPIC-0007: `master`, `crossfader` (as a property of `mixer`, not a sub-tree), and four channel sub-trees keyed `A`, `B`, `C`, `D` (see §1.5.2).
- [ ] For each channel `X` in `{A, B, C, D}`, the sub-tree `mixer.channel.X` declares writable scalar properties: `gain` (float dB, range `-inf … +12`, default `0.0`, where any value `≤ -60` dB serialises as `-inf` and maps to linear `0.0`), `eq.high` (float dB, range `-inf … +6`, default `0.0`, full CCW = true cut), `eq.mid` (same range / default / cut behaviour), `eq.low` (same), `eq.killHigh` (bool, default `false`), `eq.killMid` (bool, default `false`), `eq.killLow` (bool, default `false`), `filter` (float, bipolar range `[-1.0, +1.0]`, default `0.0`, sign positive = HPF, sign negative = LPF, magnitude = normalised cutoff travel, see §1.5.3), `fader` (float, linear range `[0.0, 1.0]`, default `1.0`), `assignA` (bool), `assignB` (bool).
- [ ] The default `assignA` / `assignB` per channel is: channel `A` → `assignA = true`, `assignB = false`; channel `B` → `assignA = false`, `assignB = true`; channel `C` → `assignA = true`, `assignB = false`; channel `D` → `assignA = false`, `assignB = true` (see §1.5.1).
- [ ] The top-level property `mixer.crossfader` is a float in `[0.0, 1.0]`, default `0.5`, where `0.0` = full A bus, `1.0` = full B bus.
- [ ] The sub-tree `mixer.master` declares the writable scalar property `gain` (float dB, range `-inf … +12`, default `0.0`, same `-inf` semantics as channel gain).
- [ ] For each channel `X`, the sub-tree `mixer.channel.X` declares read-only metering slots in the parallel `MixerMeterSnapshot` block (not in the `ValueTree`): `levelPeakL`, `levelPeakR`, `levelRmsL`, `levelRmsR` (each float, default `0.0`, range `[0.0, +inf)` in linear amplitude), and `clip` (bool, default `false`, latched on any sample ≥ 0 dBFS). The same slots exist for the master bus under the `mixer.master.*` identifier namespace.
- [ ] Metering values are addressable by the same dotted identifier path as `ValueTree` properties (e.g. `mixer.channel.A.levelPeakL`, `mixer.master.clip`), but resolve to a `MixerMeterSnapshot` atomic slot rather than to a `ValueTree` property. The identifier-to-slot mapping is documented in `Source/Features/Mixer/State/MixerIdentifiers.h` (header introduced by this PRD).
- [ ] All writable continuous parameters (`gain`, `eq.high|mid|low`, `filter`, `fader`, `crossfader`, `master.gain`) additionally expose a normalised `[0.0, 1.0]` form via a documented `MixerParam::toNormalised` / `fromNormalised` mapping. The normalised form is what the MIDI inbound router (PRD-0044) writes when applying a 14-bit or 7-bit continuous binding. Booleans are normalised as `false → 0.0`, `true → 1.0`.
- [ ] The dB-valued normalised mapping is monotonic and matches PRD-0010's mapping for `gain` so that the per-deck gain knob (now unified into channel gain by PRD-0054) remains binary-compatible: `linearGain = pow(10, dB / 20.0)`, with `≤ -60 dB` clamped to `0.0` linear and exposed as `-inf` in the display layer.
- [ ] The `filter` normalised mapping is symmetric around the center detent: bipolar value `0.0` ↔ normalised `0.5`; bipolar `-1.0` ↔ normalised `0.0`; bipolar `+1.0` ↔ normalised `1.0`. A configurable detent of `±0.02` around bipolar `0.0` is documented as the contract a UI atom (PRD-0059) must honour by snapping to `0.0`; the underlying `ValueTree` value remains continuous.
- [ ] A `MixerStateBridge` class registers as a `juce::ValueTree::Listener` on the `mixer` sub-tree and mirrors every writable scalar into a slot of the lock-free `MixerAtomicSnapshot` block. The bridge runs on the message thread only.
- [ ] The audio thread reads from `MixerAtomicSnapshot` exclusively via `std::atomic::load(std::memory_order_relaxed)` and writes to `MixerMeterSnapshot` exclusively via `std::atomic::store(std::memory_order_relaxed)`. The audio thread never touches the `ValueTree` and never takes a lock, allocates memory (`new`, `delete`, `malloc`, `std::vector::push_back`, `std::string`), or performs I/O (file, network, logging) along these code paths.
- [ ] UI components subscribe to changes via `juce::ValueTree::Listener` callbacks on the relevant `mixer.*` sub-tree. No mixer UI code polls the `ValueTree`, the audio thread, or the atomic snapshot for parameter values. Meter UI components are allowed (and required) to poll `MixerMeterSnapshot` via a `juce::Timer`; they MUST NOT use `ValueTree` listeners for meter data.
- [ ] When a deck is added at runtime (EPIC-0001 dynamic deck count), its mixer channel sub-tree is already present (the four sub-trees `A`…`D` are constructed eagerly at startup) and contains the documented defaults. No `ValueTree` structural change occurs on deck add.
- [ ] When a deck is removed at runtime, every writable property in its mixer channel sub-tree is reset to the documented defaults (gain → 0 dB, EQ → 0 dB, kills → false, filter → 0.0, fader → 1.0, `assignA`/`assignB` → the channel-letter defaults documented above), and the corresponding meter snapshot slots are reset to `0.0` / `false`. The sub-tree itself is not destroyed (see §1.5.5).
- [ ] PRD-0042 (Control Target Registry) is amended to register `mixer.channel.{A,B,C,D}.filter` as a `Continuous` target with `softTakeover = true` and an associated `MidiTargetCategory::FilterKnob`. The amendment is captured as an entry in PRD-0042's registry definition; this PRD ships alongside the registry update so that loading a mapping referencing `mixer.channel.A.filter` no longer produces an `UnknownTarget` validation error.
- [ ] PRD-0043 (Mapping Storage) is amended to include `mixer.channel.{A,B,C,D}.filter` in the bundled DDM4000 profile's required binding-target checklist, alongside the existing `gain`, `eq.*`, `eq.kill*`, `fader`, `assignA`, `assignB` entries.
- [ ] A unit test under `Tests/` (e.g. `MixerStateSchemaTests.cpp`) verifies: (a) every documented identifier resolves to an existing `ValueTree` property or `MixerMeterSnapshot` slot; (b) all defaults match this PRD; (c) writing a `ValueTree` property fires a `MixerStateBridge` update into the matching atomic within the same message-thread call stack; (d) per-channel `assignA`/`assignB` defaults are as specified; (e) deck removal resets the sub-tree to defaults without destroying it.
- [ ] No DSP is implemented by this PRD: there is no EQ, filter, gain stage, crossfader curve, or metering computation. The schema is wired to placeholders that downstream PRDs (PRD-0053…PRD-0058) consume.
- [ ] No UI components are implemented by this PRD. UI is owned by PRD-0059 / PRD-0060.
- [ ] The audio-engine summing stage is not modified by this PRD. The signal-flow refactor is owned by PRD-0053.

## 1.5. Grey Areas

### 1.5.1. Default A/B Crossfader Assignment per Channel

The Epic does not prescribe the default crossfader-assign pattern for the four channels. Three plausible defaults exist: (a) all channels assigned to both A and B (centered / through, meaning the crossfader is effectively bypassed until the DJ explicitly assigns); (b) all channels assigned to neither (silent until configured); (c) alternating A / B / A / B across channels A, B, C, D to mirror the canonical two-deck-on-each-side layout of a four-channel DJ mixer (DJM-900, DDM4000).

**Resolution:** Adopt pattern (c): channels A and C default to `assignA = true, assignB = false`; channels B and D default to `assignA = false, assignB = true`. This matches the muscle-memory layout of every four-channel DJ mixer the target user has used: odd-numbered channels live on the left side of the crossfader, even-numbered channels on the right. Pattern (a) silently nullifies the crossfader, which would confuse a new user dragging the crossfader and hearing nothing change. Pattern (b) requires explicit setup before any audio routes through the mixer at all, which contradicts the goal of a fluid workflow. The DJ remains free to reassign at any time; the default merely makes a freshly launched four-deck session immediately playable.

### 1.5.2. Channel Sub-Tree Key: Deck Letter vs Numeric Index

The mixer channels could be keyed by deck letter (`A`, `B`, `C`, `D`) — matching the deck identifiers used throughout PRD-0001, PRD-0042, and every bundled MIDI mapping — or by a stable numeric index (`0`, `1`, `2`, `3`) which is more conventional for collections that may grow.

**Resolution:** Key by deck letter. The MIDI binding registry (PRD-0042) and every mapping file already encodes mixer paths as `mixer.channel.A.*` through `mixer.channel.D.*`. Switching to numeric indices would require either renaming every binding target (breaking PRD-0042's stability guarantee and invalidating community mappings) or maintaining a lossy letter-to-index translation layer with no upside. The maximum channel count is fixed at four by EPIC-0001 and EPIC-0007 (Pioneer / Allen & Heath / DDM4000 form factor), so the "what if we grow past 26 channels" objection does not apply. Migration safety: because the letter keys match the existing PRD-0042 target identifiers, no migration step is needed for any pre-existing mapping file or registry entry; this PRD is purely additive on top of those documents.

### 1.5.3. Filter Storage: Bipolar `[-1, +1]` vs Unipolar `[0, 1]` with Center Detent

The per-channel filter knob is one physical control covering two filter modes (HPF clockwise, LPF counter-clockwise) with a bypass at 12 o'clock. It can be modelled as a unipolar `[0, 1]` value where `0.5` means bypass, or as a bipolar `[-1, +1]` value where `0.0` means bypass, sign indicates filter mode, and magnitude indicates cutoff travel.

**Resolution:** Store the filter value as bipolar `[-1.0, +1.0]` with `0.0` as the explicit bypass center. The DSP block (PRD-0056) needs to know which filter (HPF or LPF) is active and how far from bypass the cutoff sits; both pieces of information fall out of the bipolar representation in a single read (sign for mode, `|value|` for normalised cutoff). The unipolar form forces every consumer to subtract `0.5` and re-introduce a sign anyway. The center detent is a UI-layer behaviour (snap to bipolar `0.0` within `±0.02`, documented in §1.4) and is intentionally not enforced inside the `ValueTree`, so MIDI controllers and automation can pass through the detent zone continuously without being quantised. The normalised `[0.0, 1.0]` form exposed to the MIDI inbound router (PRD-0044) maps `bipolar -1.0 → 0.0`, `bipolar 0.0 → 0.5`, `bipolar +1.0 → 1.0`, which is what every standard MIDI bipolar encoder already emits.

### 1.5.4. Metering Values: Inside the `ValueTree` vs Parallel Atomic Snapshot

Level meters update at the UI refresh rate (30–60 Hz per channel for peak/RMS L/R plus a clip latch — roughly 5 scalar updates per channel per frame, times four channels plus the master bus). If those updates were stored as `ValueTree` properties, every frame would fire dozens of `valueTreePropertyChanged` callbacks on the message thread and would force the audio thread to either acquire a lock to mutate the `ValueTree` (forbidden by `AGENTS.md`) or to marshal updates through an extra lock-free FIFO that gets drained into the `ValueTree` on the message thread (added complexity, listener storms).

**Resolution:** Metering values live in a parallel `MixerMeterSnapshot` block of `std::atomic<float>` / `std::atomic<bool>` slots, addressed by the same dotted identifier path used for `ValueTree` properties but resolved through a separate lookup table. The audio thread writes meter values with `std::atomic::store(std::memory_order_relaxed)`; UI meter components read them with `std::atomic::load(std::memory_order_relaxed)` from a `juce::Timer` callback. This keeps the audio thread fully lock-free and allocation-free, eliminates listener storms on the message thread, and decouples meter UI refresh rate from parameter change notifications. The cost — losing `ValueTree` change notifications for meter values — is intentional, because meters are pull-based by nature.

### 1.5.5. Deck Removal: Keep, Reset, or Destroy the Mixer Channel Sub-Tree

When a deck is dynamically removed (4 → 3 → 2 decks, or 2 → 1), the question is whether its mixer channel sub-tree should be (a) kept as-is (state persisted, so re-adding the deck restores its mixer settings), (b) reset to defaults (state cleared, re-add starts fresh), or (c) destroyed (sub-tree removed from the `ValueTree`).

**Resolution:** Keep the sub-tree structurally, but reset its writable properties to defaults on deck removal. Destroying the sub-tree (option c) would force every `ValueTree::Listener` and every cached MIDI binding to re-resolve its path on every deck re-add, would require PRD-0042's compile-time registry to tolerate runtime target removal (it does not — targets are `constexpr`), and would create surprising behaviour for MIDI controllers whose mappings statically reference channel D. Keeping state as-is (option a) is appealing for "undo my accident" but conflicts with the user's mental model: removing a deck is a deliberate scoping action, and the next user who adds a fourth deck expects a clean channel strip, not someone else's leftover EQ kills. Reset (option b) gives the deterministic behaviour the user expects while preserving the structural invariant that all four channel sub-trees always exist for the entire application lifetime. The same reset applies to the meter snapshot slots for that channel.

### 1.5.6. Identifier Path Style: Dotted String vs Hierarchical `juce::Identifier`

`juce::ValueTree` natively addresses children via `juce::Identifier` per level (`tree.getChildWithName("mixer").getChildWithName("channel").getChildWithName("A").getProperty("gain")`), while PRD-0042 and every bundled mapping file already use the dotted string form `mixer.channel.A.gain` for MIDI binding targets.

**Resolution:** Use both, with a single translation point. Internally, the `ValueTree` is structured hierarchically with one `juce::Identifier` per level (`mixer` → `channel` → `A` → property `gain`). Externally — in the MIDI registry, in mapping files, in logs, in the meter-snapshot lookup table, and in this PRD — identifiers are written as dotted strings. A header `Source/Features/Mixer/State/MixerIdentifiers.h` defines a single `constexpr` mapping between the dotted form and the hierarchical path, so that both representations stay in lockstep and there is exactly one place to amend when adding a parameter.
